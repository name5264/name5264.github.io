블로그를 만들게 된 이유

사실은 원래 내 정보를 남한테 공유를 하는 것을 좋아하진 않는다.

내가 아는 정보가 틀릴수도 있고, 내가 이해한 방법이 다른 사람에게는 맞지 않을 수도 있을 것이라 생각하기 때문이다.

하지만, 어느 블로그에서 블로그를 해야하는 이유를 자세히 적은 것 1 을 보고 블로그를 해볼까? 라는 생각이 들었다.

그 뒤로 블로그를 시작할려고 티스토리와 velog 중에서 고민을 하다, 재밌는 생각이 떠올랐다!

내가 한번 블로그를 만들어볼까?

꽤나 좋은 생각 같아서 리엑트 공부겸 만들기를 시작했다.

후보 1 : CRA + EXPRESS

원래는 프론트 엔드로 react + typescript를 생각하고, 백엔드로 express.js + typescript를 생각했지만 SSG를 위해 다른 방법을 찾았다.

후보 2 : Jekyll

가장 많이 쓰이는 생성기고 커뮤니티가 활발하지만, 낮선 Ruby언어 때문에 다른 대안을 찾아보았다.

후보3 : Gatsby + styled-components

그나마 익숙한 React 기반에 mdx를 지원하고, 넉넉한 plugin과 쉬운 검색때문에 최종적으로 Gatsby를 선택하게 되었다.

styled-components를 사용해 스타일에 대한 부분도 깔끔하게 정리 하기로 했다.

나한텐 너무 어려운 Gatsby

react를 공부하기도 했고, 잘 써먹고 있긴 하지만, gatsby 같은 프레임 워크와 같이 사용하니 이야기가 달라졌다.

plugin 적용이 마음대로 되지 않고, 배포에도 문제가 주르륵 뜨고...

특히 만악의 근원이 map() 이였다.

만악의 근원 .map()

// 만악의 근원... 너때문에 빌드가 안돼..
data.map(v => {
  const title = v.frontmatter.title

  return (
    <Card
      title={title}
      key={title}
      to={v.slug}
      description={v.frontmatter.description}
    />
  )
})

이 코드는 내 스트레스의 결정적 원인인 코드를 그대로 옮겨 온 것이다. 이게 왜 내 스트레스의 원인일까?

바로 이거때문에 빌드가 안되었다...

사건의 전말

일단 gatsby 프로젝트를 생성했다.

$ npx gatsby new blog https://github.com/gatsbyjs/gatsby-starter-blog

그리고 pages/index.js 안에 내용을 전부 지우고, 새로 코드를 짜 집어 넣었다.

분명 gatsby develop 에서는 잘 작동이 되던 코드다!

하지만 왜 gasby build 에서만...

[                            ]   0.000 s 0/4 0% Building static HTML for pages

이 부분에서 계속 20분이 지나도 진행될 기미가 안보이는 것이다!

그래서 코드 여러 부분을 들춰보고 뜯어보고 고쳐봤더니..

data.map(v => {
  const title = v.frontmatter.title

  return (
    <Card
      title={title}
      key={title}

딱 이 부분을 지우니 빌드가 되는 것이다!

이 부분을 다시 살리고 뭘 수정하고 코드를 다시 짜도 빌드가 계속 되지 않는다...

이 부분을 포기하기에는 게시글을 띄우는 코드라 포기할 수도 없었다.

그렇게 눈물을 머금고 프로젝트를 지우고 새로 프로젝트를 열었다.
